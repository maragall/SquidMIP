# IMA-213 — Rename cutover: SquidMIP -> SquidHCS

_slug: rename-squidhcs_

## Why

The repo, the Python package, and the GitHub URL still carry the `SquidMIP` /
`squidmip` name. The tool's output format is already `.hcs` and the viewer is
already called the HCS viewer, so the package name is the last thing out of step.

**Sequencing was re-decided during eng review (2026-07-20). This no longer lands last.**

The original spec deferred this ticket because "Nick depends on the current repo
URL." That premise does not hold: renaming a GitHub repository installs a
permanent redirect, so Nick's existing clone, his `git pull`, and any
`pip install git+https://github.com/maragall/SquidMIP` keep working untouched.
The repo-URL half of the rename is free.

The half that costs money is the package directory move, and its cost is merge
conflicts.

**Corrected branch state (outside voice caught an error in the first pass — the
original review compared local refs to local `main` instead of `origin/main`):**

- 17 branches exist on `origin`; **11 are 1 commit ahead of `origin/main`**
  (ima-176, 186, 192, 206, 216, 221, 223, 225, 226, 227, 234).
- Every one of those commits touches `TODOS.md` and nothing under the package
  directory. `TODOS.md` carries 3 `squidmip` references, so the live conflict
  surface today is **one markdown file**, not package code.
- The remaining ~30 local worktree branches are empty placeholders at `1504c05`.

So "conflicts with nothing" was overstated. The accurate version: the conflict
surface today is one line-level conflict in one markdown file, and it grows every
time a ticket lands real code under the package directory. IMA-205..IMA-234 are
queued to do exactly that. Landing the rename now is cheap; landing it last means
rebasing a whole-directory rename through however much of that work has shipped.

The direction holds. The quantification in the first pass did not.

## Scope

Two renames, one PR:

1. **GitHub repo** `maragall/SquidMIP` -> `maragall/SquidHCS`. GitHub's automatic
   redirect covers the old URL; no code change needed for it to keep working.
2. **Python package** `squidmip/` -> `squidhcs/`, plus every live reference:
   imports, `pyproject.toml`, `environment.yml`, both CI workflows, the scripts,
   README, live docs, and the two checked-in `.html` artifacts.

**Verified inventory: 216 occurrences across 42 tracked files** (`git ls-files |
xargs grep -o`). The first review pass reported 186/38 — that grep used
`--include` filters that skipped `scripts/mip-tool.bat` and counted matching
lines rather than occurrences. Files the first pass missed entirely:
`scripts/mip-tool.bat` (5), `COMMITS-EXPLAINED.md`, `TODOS.md` (3),
`docs/DESIGN.md`, `docs/DESIGN-STATUS.md` (12), `docs/SCOPE-video-player.md` (6).

Plus the migration and verification work the review added (see Decisions).

### Explicitly NOT renamed

- **The product name stays "MIP tool"** (D7). The Desktop shortcut label, the
  README title, and the app window title are unchanged. The tool really does
  perform maximum intensity projection; `MIP` throughout the docs is domain
  vocabulary, not the package name, and must survive intact.
- **Historical design docs keep their `squidmip` text** (D6).

## The break this ticket must not cause

Nick does not reinstall the tool. He clicks a Desktop shortcut, and README:21
tells him to update with `git pull`.

```
scripts/Setup-Windows.ps1
  :13   $AppName = "MIP tool"
  :14   $Module  = "squidmip._viewer"
  :40   $venv = Join-Path $env:LOCALAPPDATA "squidmip\venv"
  :54   & $vpy -m pip install -e ($repo + "[gui]")     <-- EDITABLE
  :67   $sc.Arguments = "-m " + $Module
```

The shortcut invokes the old venv's python with `-m squidmip._viewer`. The install
is editable, so the moment `git pull` brings down a tree where the package is
`squidhcs/`, the module is simply gone.

```
BEFORE                                    AFTER `git pull`, NO MIGRATION
Desktop\MIP tool.lnk                      Desktop\MIP tool.lnk
  -> %LOCALAPPDATA%\squidmip\venv\          -> %LOCALAPPDATA%\squidmip\venv\
       Scripts\python.exe -m squidmip._viewer    Scripts\python.exe -m squidmip._viewer
         |                                          |
         v  editable -> repo\squidmip\              v  editable -> repo\squidhcs\
       app opens                                  ModuleNotFoundError: No module
                                                  named 'squidmip'   <-- Nick's tool is dead
```

The original acceptance oracle ("fresh clone installs and launches") does not
cover this, because Nick is not doing a fresh clone.

### 1. Package rename (mechanical)

- `git mv squidmip squidhcs` (use `git mv` so `git log --follow` survives).
- Rewrite imports across 13 package modules, 15 test files, and 3 scripts.
- `pyproject.toml`: `name`, `[project.scripts]` (`squidhcs`, `squidhcs-view`),
  `[tool.hatch.build.targets.wheel] packages`.
- `squidhcs/__init__.py`: module docstring examples + `__all__` surface.
- `environment.yml`: `name: squidhcs`, comment block commands.
- `scripts/mip-tool.bat`: **delete it** (D11). It `conda activate squidmip`s and
  calls `squidmip-view`, but `Setup-Windows.ps1:1-8` documents a plain venv and
  says "NO conda" outright. Renaming the file would preserve two contradictory
  install stories; deleting it leaves one.
- `scripts/hcs_viewer_entry.py`, `scripts/smoke_import.py` (and D9).
- README + `docs/hcs-viewer-quickstart.md` command examples.
- `docs/architecture.html`, `docs/plate-view.html` — hand-edit; no generator
  exists in the repo (verified by grep).

**Guardrail:** `MIP` as domain vocabulary must not be touched. Never run a
case-insensitive replace on `mip`. Replace only the exact tokens `squidmip`
and `SquidMIP`.

### 2. Windows migration (D3) — `scripts/Setup-Windows.ps1`

```
Setup-Windows.ps1 (new logic)
  |
  ├─ old venv at %LOCALAPPDATA%\squidmip\venv exists?
  │    ├─ yes ─> build new venv at ...\squidhcs\venv
  │    │         install -e repo[gui]
  │    │         rewrite Desktop "MIP tool.lnk" -> new vpy, -m squidhcs._viewer
  │    │         remove the stale old-path shortcut (no duplicate icons)
  │    │         leave the old venv on disk (non-destructive; user may delete)
  │    └─ no  ─> normal fresh install at ...\squidhcs\venv
  |
  └─ idempotent: safe to re-run any number of times
```

`$AppName` stays `"MIP tool"` (D7), so the icon label Nick recognises is
unchanged — only its target moves.

**Two points the outside voice added:**

- The old venv is left on disk deliberately (non-destructive), but it is several
  hundred MB of PyQt5 + tensorstore + ndviewer_light. Print its path and a
  one-line "safe to delete" message rather than leaving it silently orphaned.
- Nick's *local clone folder* keeps its `SquidMIP` name — a GitHub rename does
  not rename his directory, and the editable install path stays valid, so the
  migration does not need to handle a folder move. But if he ever follows the
  updated README and clones `SquidHCS` fresh, he ends up with two folders while
  the editable install still points at the old one. Say this explicitly in the
  README migration note.

### 3. CI hardening (D5)

- `build.yml`: `--collect-submodules squidmip` -> `squidhcs`.
- `build.yml`: `if-no-files-found: warn` -> `error`.
- `build.yml`: add a step after `freeze` asserting the built binary exists and
  is non-empty, so an empty freeze fails the job instead of warning.

### 4. Tests (regression rule — mandatory)

- **`tests/test_rename_guard.py` (CRITICAL).** Walk `git ls-files` and assert
  zero occurrences of `squidmip` / `SquidMIP`, with an explicit allowlist for
  the historical docs preserved under D6. One test covers `.bat`, `.ps1`,
  `.yml`, `.html`, README, and every module at once.
  *Why critical:* the existing suite imports the package directly, so it passes
  even if every non-Python reference still points at the dead name.

  **Anti-tautology requirement (outside voice).** A guard whose allowlist the
  implementer edits to make their own test pass proves nothing. So the allowlist
  must be a frozen literal in the test file, and the test must additionally
  assert the allowlist has not grown — hardcode the exact expected file set and
  per-file occurrence counts, and fail if either changes. The allowlist covers
  exactly the D6 docs and nothing else; `TODOS.md`, `docs/DESIGN.md` and
  `docs/SCOPE-video-player.md` are **live** files and get renamed, not allowlisted.
- **`tests/test_entry_points.py` (CRITICAL).** `subprocess` the installed
  `squidhcs --help`, `squidhcs-view --help` (`QT_QPA_PLATFORM=offscreen`), and
  `python -m squidhcs --help`; assert exit 0.
  *Why critical:* D4's clean cut makes the console-script names load-bearing,
  and no test in the repo has ever resolved an entry point.
- **`build.yml` Windows migration job (D8).** Fabricate an old
  `%LOCALAPPDATA%\squidmip\venv` and a stale shortcut, run `Setup-Windows.ps1`,
  assert the new venv exists, the app imports, and the old shortcut is gone.

### 5. GitHub repo rename

- Rename in GitHub settings; confirm the redirect serves the old URL.
- Update `git remote set-url` in the local worktrees (optional; redirect covers it).
- Verify `pip install git+https://github.com/maragall/SquidMIP` still resolves.

## What already exists (reused, not rebuilt)

- `ci.yml` clean-room PyPI-only install — already proves the package builds and
  imports off-machine. Reused as-is; the rename needs no new install harness.
- `scripts/smoke_import.py` — already the per-OS dependency-import check. Reused;
  only its module list changes (plus D9).
- `build.yml` Windows runner — already provisioned. D8's migration job reuses it
  rather than adding new infrastructure.
- `Setup-Windows.ps1` already rewrites the Desktop shortcut on every run, so D3's
  migration is an extension of existing behavior, not a new mechanism.
- GitHub's repo-rename redirect — free platform behavior, replaces the custom
  "old-name redirect" the original spec implied we had to build.

## NOT in scope

- **A `squidmip` compat shim package** — rejected at D3. Keeping the old name
  importable defeats the ticket and does not fix the venv path, which is the
  actual break.
- **Old console-script aliases** — rejected at D4. Clean cut instead.
- **Renaming the product to "SquidHCS"** — rejected at D7. Repo and package only.
- **Rewriting historical eng-review docs** — rejected at D6. Annotated instead.
- **A conda-path CI job** (`environment.yml` create + run) — the venv path in
  `Setup-Windows.ps1` is the documented install route and gets a real CI job at
  D8; the conda route is legacy and covered by the D6 grep guard for staleness.
- **Fixing `docs/*.html` generation** — no generator exists; hand-editing two
  files is cheaper than building one. Captured as a TODO instead.
- **Version bump** — the package is git-install only (no PyPI publish workflow
  exists), so `0.1.0` carries no resolution semantics. Left alone.

## Worktree parallelization strategy

Sequential implementation, no parallelization opportunity. Every work item
touches the same package directory rename, and items 2-4 depend on item 1
having landed. Splitting across worktrees would create the exact merge conflict
this ticket's timing is designed to avoid.
