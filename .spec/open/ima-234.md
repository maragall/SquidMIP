# IMA-234 — Odon remote benchmark

_slug: odon-remote-benchmark_

## Why

The web-perf proof: does an OME-zarr plate stay usable when it is served over HTTP
instead of read off local disk? The answer feeds IMA-200 (cloud-host minerva /
vitessce / Allen Cell Explorer). Odon (`alexcoulton/odon`) is the renderer under
test because it reads OME-Zarr local *and* remote through one viewport-tile engine,
so the same binary can drive both arms.

## Scope

Build an instrumented static HTTP server that serves a `plate.ome.zarr` and records
every chunk request, so the local→remote cost can be measured from the server side
rather than guessed by eye. Actually launching Odon against the two arms is a
documented human procedure, not part of the automated build (see Decision 1).

## ✅ PRE-FLIGHT GATE — RESOLVED 2026-07-20, build unblocked

The outside voice raised a possible showstopper: SquidMIP writes NGFF **v0.5 on
zarr v3**, and many viewers read only NGFF 0.4 / zarr v2. Resolved by reading
Odon's `Cargo.toml` (v0.1.5):

```
zarrs             = { version = "0.23.3", features = ["zlib"] }   # Zarr V2/V3 for Rust
zarrs_http        = "0.3.1"                                       # HTTP store
zarrs_object_store = "0.6.2"
object_store      = { version = "0.13", features = ["aws"] }      # real S3
tokio             = { version = "1.49.0", features = ["rt-multi-thread", ...] }
lru               = "0.16.3"
```

**Check 1 — zarr v3: PASS.** `zarrs` is the Rust Zarr **V2/V3** crate, so the store
and array layer reads zarr v3 natively. The ticket is not void.

**Check 2 — timing signal: FAIL (as suspected).** No FPS counter, verbose mode, or
debug log is documented. This *confirms* Decision 2 rather than undermining it: there
is nothing client-side to read, so the server access log genuinely is the only
instrument available.

**Three consequences that change the plan:**

- **Residual risk is now narrow and cheap.** `zarrs` handles the *zarr* layer; the
  *NGFF v0.5 metadata* layer (`multiscales`/`omero` under `attributes.ome` in
  `zarr.json`, rather than 0.4-style `.zattrs`) is parsed by Odon itself — it pulls
  `serde_json`, not an OME metadata crate. So the risk drops from "can't read the
  store at all" to "may look for 0.4-style metadata keys." That is a one-run smoke
  check in the manual arm, not a blocker. Note this is the same failure class the
  repo already documented for ndviewer_light.
- **A real bucket is now nearly free** (`object_store` + `aws` already compiled in),
  so it is added to the manual procedure rather than argued against — see Decision 3'.
- **`tokio` multi-thread + `lru`** confirm Odon fetches concurrently and caches tiles,
  which makes Decision 4 (ThreadingHTTPServer) load-bearing and means cache state must
  be controlled between runs.

## ⚠ Known limitation, now resolved by reframing

The ticket's headline is a *local→remote delta*. The locked metrics (Decision 2)
exist **only in the HTTP arm** — local-file reads are invisible without dtrace. The
harness therefore cannot compute that delta.

**Resolution: report a latency-sensitivity curve, not a delta.** The deliverable is
`first_view_ms` and `req_per_s` across 0 / 25 / 100 ms of injected latency plus the
real-bucket arm, with the local-file arm contributing one stopwatch time-to-usable-view
as a floor reference. Both numbers get reported in their own units and clearly labelled;
the word "delta" is dropped from the deliverable because the design cannot produce one
honestly.

## Decisions locked (plan-eng-review, 2026-07-20)

**1. Split: harness is built, measurement is performed by hand.**
`which odon` returns "odon not found" on this machine, and IMA-212 — the bridge that
writes the zarr and launches Odon — is still in Linear Backlog with no spec file. An
unattended build cannot install a GPL GUI binary, drive it, or read a framerate off a
screen. So the automated deliverable is the harness plus its tests, which must pass
with **no odon binary present**. The measurement procedure is written down (below) and
run by a human afterwards.

**2. Metrics are defined server-side, from the access log.**
Odon is a black box; there is no hook into its render loop. But the two arms are
asymmetric in a useful way: in the HTTP arm every chunk Odon reads *is* an HTTP request
this server logs, with timestamps, paths, statuses, and byte counts. In the local-file
arm those reads are invisible without dtrace. The HTTP server is therefore not just
transport, it is the only instrument available. "First paint" and "framerate" get
concrete server-side proxies (defined under Metrics).

**3. Latency injection, because localhost is not "remote".**
Loopback round-trip is ~0.05 ms with effectively unlimited bandwidth; a real object
store is 20–100 ms away. Local-file vs localhost-HTTP measures only HTTP framing and
per-chunk syscall overhead, which would show a small delta and support the wrong
conclusion — that remote is nearly free. A `--latency-ms` flag delays each response so
the run sweeps 0 / 25 / 100 ms and reports a latency-sensitivity curve instead of one
flattering number.

**3'. Add a real-bucket arm (added after the pre-flight gate).**
The outside voice argued a real S3/R2/GCS bucket beats simulated latency, because
`time.sleep` models neither TLS handshakes, keep-alive, connection pooling, nor the
bandwidth-delay product. The gate check shows Odon already links `object_store` with
the `aws` feature, so pointing it at a real bucket costs **zero code on our side** —
it is a URL change. So both, not either: keep the local harness (reproducible, free,
sweepable, and the only source of the per-level and per-status breakdown, which a
bucket will not hand you) and add one real-bucket run to the manual procedure as the
ground-truth check that the simulated curve is honest. If the bucket number lands far
off the curve, trust the bucket and say so.

**4. `ThreadingHTTPServer`, not the default `HTTPServer`.**
The default handles one request at a time, so a tile viewer's parallel chunk fetches
queue behind each other and the measured delta becomes the harness's own serialization
— worse the more parallel the viewer is, which is backwards. It also breaks Decision 3:
on a single-threaded server, per-request sleeps stack instead of overlapping, so a
25 ms injection becomes 25 ms × queue depth. Stdlib since Python 3.7, no new dependency.

**5. Tests build a real fixture plate through the shipping writer.**
Reuse `tests/conftest.py::squid_dataset` → `squidmip.open_reader` → `squidmip.write_plate`
so the harness is exercised against the actual chunk layout and pyramid structure
SquidMIP emits, not a hand-made approximation that can silently drift. Odon-dependent
cases skip the way `tests/test_performance.py:27` already does.

**6. Harness lives in `scripts/`, not the shipped package.**
`squidmip/` is what gets installed for users; `scripts/` already holds non-product
helpers (`hcs_viewer_entry.py`, `smoke_import.py`). A third-party-viewer eval harness
should not ship to every MIP-tool install. Tests import it via a `sys.path` insert in
`tests/conftest.py`.

## Design

```
                       ┌─────────────────────────────────────────┐
                       │  scripts/odon_bench.py                  │
                       │                                         │
   plate.ome.zarr ────►│  serve(root, port, latency_ms)          │
   (from write_plate)  │    └─ ThreadingHTTPServer               │
                       │         └─ ChunkLogHandler              │
                       │              ├─ sleep(latency_ms)  [D3] │
                       │              ├─ SimpleHTTPRequestHandler│
                       │              └─ log.record(...)    [D2] │
                       │                     │                   │
                       │                     ▼                   │
                       │              RequestLog (thread-safe)   │
                       │                     │                   │
                       │       summarize() ──┴──► format_report()│
                       └─────────────────────────────────────────┘
                                 ▲                        │
                                 │ HTTP GET chunks        ▼
                          ┌──────┴──────┐          metrics report
                          │    Odon     │          (markdown/JSON)
                          │  (external, │
                          │   by hand)  │
                          └─────────────┘

   ARM A: odon <path>            → no server, no log (stopwatch only)
   ARM B: odon http://…  --latency-ms 0    → full server-side log
   ARM C: odon http://…  --latency-ms 25   → same, +25ms per response
   ARM D: odon http://…  --latency-ms 100  → same, +100ms per response
```

### Metrics (server-side, from the access log)

| Metric | Definition |
|---|---|
| `ttfb_ms` | first request received → first response fully written |
| `first_view_ms` | first request → start of the first idle gap ≥ `--idle-gap-ms` (default 500). This is the first-paint proxy: the opening view has settled when the viewer stops asking for chunks. |
| `req_per_s` | requests served ÷ active (non-idle) wall time. The framerate proxy during a scripted pan/zoom. |
| `peak_concurrency` | max in-flight requests. Also proves Decision 4 is working. |
| `bytes_total` | sum of response body sizes |
| per-status | 200 vs 404 counts. 404s are free on loopback but cost a full round trip at 100 ms, so probe-heavy clients degrade non-linearly. |
| per-level | requests bucketed by pyramid level parsed from `{row}/{col}/{fov}/{level}/…`. Answers the open TODOS.md question about whether anything reads IMA-184's pyramid. |

## Test plan

```
CODE PATHS                                          USER FLOWS
[+] scripts/odon_bench.py                           [+] Operator runs the benchmark
  ├── RequestLog.record()                             ├── [GAP] serve → URL printed → Ctrl-C → report
  │   ├── [GAP] concurrent record from N threads      ├── [GAP] Ctrl-C before any request → "no requests"
  │   └── [GAP] records status + size + level         └── [GAP] bad zarr path → clear error, exit != 0
  ├── summarize()
  │   ├── [GAP] empty log → no ZeroDivisionError    [+] Error states
  │   ├── [GAP] idle-gap → first_view_ms              ├── [GAP] port already in use → clear message
  │   ├── [GAP] never idle → first_view = total       └── [GAP] unparseable chunk path → "other", no crash
  │   └── [GAP] per-status / per-level buckets
  ├── ChunkLogHandler
  │   ├── [GAP] latency_ms=0 → no sleep call
  │   ├── [GAP] latency_ms=50 → single req ≥ 50ms
  │   ├── [GAP] [CRITICAL] 2 concurrent 50ms reqs finish < 100ms  (proves D3+D4 together)
  │   └── [GAP] 404 for missing chunk → logged, not fatal
  ├── serve() contextmanager
  │   └── [GAP] real fixture plate.ome.zarr, multi-threaded urllib fetch
  └── main()
      ├── [GAP] --help exits 0
      └── [GAP] missing --zarr → exit != 0

COVERAGE: 0/16 paths tested (new code)  |  all GAPs are in-scope for this build
```

The **critical** test is the concurrency one: two simultaneous requests with
`--latency-ms 50` must complete in well under 100 ms. If they take ~100 ms, the server
is serializing and every number the harness produces is a harness artifact. That single
test guards Decisions 3 and 4 at once.

**Assert it loosely.** A tight wall-clock bound flakes on a loaded machine, and the repo
already handles this with generous ceilings in `tests/test_performance.py`. Use a wide
margin (e.g. total < 85 ms for two 50 ms requests) — the failure being caught is
serialization, which doubles the time, not a few milliseconds of scheduler jitter. An
unattended agent will burn iterations fighting a tight bound.

**What these tests do and do not prove.** The end-to-end test fetches chunks with a
Python loop, so idle-gap detection is finding the shape of the test's own traffic. That
validates the log analyzer, not viewer performance. Stated plainly so the green suite is
not mistaken for a benchmark result — the benchmark result requires the manual Odon arm.

### Failure modes

| Failure | Test? | Handled? | Visible? |
|---|---|---|---|
| Zero requests (Odon never launched) | yes | yes — report says "no requests" | clear |
| Port already bound | yes | yes — explicit error, non-zero exit | clear |
| Log race under concurrency | yes | yes — lock around accumulate | would otherwise be **silent** |
| Chunk path not `row/col/fov/level` | yes | yes — bucketed "other" | clear |
| Server serializes requests | yes (critical) | n/a — correctness bug | would otherwise be **silent** |

No critical gaps: every silent-failure mode above has a test.

## Acceptance / oracle

**Automated (this build):**
- `pytest tests/test_odon_bench.py` passes with **no odon binary installed**.
- The full existing suite still passes (`pytest -m "not integration"`), and
  `python -c "import squidmip"` succeeds — no regression.
- `python scripts/odon_bench.py --zarr <fixture>/plate.ome.zarr --latency-ms 25`
  serves, logs, and prints a report containing every metric in the table above.

**Manual (human, after the build):**
1. Install Odon v0.1.5 (macOS arm64, runs native on M4).
2. Produce a plate: `squidmip <acquisition> <out>` → `<out>/plate.ome.zarr`.
3. **Smoke check first (residual risk from the gate):** open the plate in Odon at all.
   If it fails to parse, the cause is almost certainly NGFF v0.5 metadata layout
   (`attributes.ome` in `zarr.json` vs 0.4-style `.zattrs`), not zarr v3. Record the
   error and stop — that finding is worth more than the benchmark.
4. Arm A — local: launch Odon on the path directly, stopwatch to first usable view.
5. Arms B/C/D — served: `python scripts/odon_bench.py --zarr <out>/plate.ome.zarr
   --latency-ms {0,25,100}`, point Odon at the printed URL, run the *same* pan/zoom
   sequence each time, Ctrl-C, save the report.
6. Arm E — real bucket: upload the plate to S3/R2/GCS, point Odon at the bucket URL
   (Odon links `object_store` with `aws`, so no code needed). This is the ground truth
   the simulated curve is checked against.
7. **Restart Odon between arms.** It links `lru`, so it caches tiles in-process; reusing
   one session would serve the second arm from memory and fabricate a speedup.
8. Report the **curve**, not a delta: `first_view_ms` and `req_per_s` across arms B–E,
   with arm A's stopwatch as a floor reference in its own units. Run each arm 3× and
   take the median. OS page cache stays warm across B–D by design — that isolates
   transport rather than disk, which is the intended comparison.

## NOT in scope

- **The IMA-212 bridge** (samplesheet / `project.json` writer, Odon launcher) — its own
  ticket, its own review; absorbing it would double a 2-point ticket.
- **Real network shaping** (`pfctl` / dummynet for true RTT + bandwidth caps) — needs
  sudo and machine-wide rules an unattended build cannot safely set up or clean up.
  Injected per-request latency is the tractable substitute.
- **Screen-recorded frame analysis** for true rendered framerate — capture tooling,
  thermal variance, and macOS permissions outweigh a 2-point eval.
- **Bandwidth throttling / TLS / connection limits** — latency is the dominant term and
  the only one modelled here. Stated so the result is not over-read.
- **Distribution** — no new shipped artifact; the harness stays out of the wheel per
  Decision 6, so there is no build/publish pipeline to add.

## Outside voice — disposition

An independent reviewer raised 10 findings. Disposition after the pre-flight gate:

**Folded in (5):**
1. NGFF v0.5 / zarr v3 feasibility → ran the check, PASS, documented above.
2. "Local→remote delta is not computable" → correct; deliverable reframed to a curve.
3. "Use a real bucket" → adopted as arm E (Decision 3'), since Odon already links
   `object_store`+`aws` so it costs no code.
4. Flaky tight timing bound → loosened per the repo's existing convention.
5. "Tests only prove the analyzer analyzes logs" → stated plainly in the test plan so a
   green suite is never mistaken for a benchmark result.

**Measurement caveats to apply when reporting (2):** idle-gap conflates "painted" with
"stopped prefetching" (viewers trickle low-priority tiles after paint), and zarr v3
metadata discovery means TTFB may be dominated by a `zarr.json` crawl of many tiny
GETs — a store-layout artifact, not image-serving cost. Neither invalidates the design;
both must be named in the write-up.

**Deferred to IMA-200, not this ticket (3):**
- **Odon may not generalize to vitessce.** Vitessce is a browser client with a different
  fetch engine, HTTP/2, and web workers, and browser devtools instruments first-paint
  natively with no harness at all. IMA-234 is scoped as an *Odon* eval per IMA-212; the
  browser-client measurement is separate work IMA-200 needs on its own.
- **Cost metrics.** Egress and storage per plate view. Cloud hosting is a spend decision
  and no arm here produces a dollar figure.
- **Multi-user concurrency.** A hosted plate serves N scientists at once; every arm here
  is single-client.
