# IMA-176 — SquidMIP (epic) · MIP tool

High-throughput z-stack max projection for Squid well-plate acquisitions. This is the **foundation tool (MIPtool)**.

**Thesis:** the projection is trivial (`np.max` over z, identical to FIJI). The value is deleting FIJI's manual, format-blind workflow: parse the whole plate from metadata, project in one command, hand back an easy-to-view result. Same metadata drives batch AND viewing.

**Decisions (anchored to the requirements):**
- MIP is the method; **ship MIP first, design the projector pluggable** so EDF/best-focus and mean drop in later (requirement — not de-scoped, not built now). Seam is a single `project()` function signature, not a plugin registry.
- CPU, I/O-bound, parallel across wells, streaming; no GPU.
- Reuse `tilefusion.open_reader` for format recognition — the value over FIJI.
- Output = TIFF (flavor TBD with Nick) + navigable OME-zarr; "easy to view" = montage or ndviewer.

**Relationship to squid2minerva:** squid2minerva is a **separate product built ON TOP of MIPtool** (adds Minerva-export/story). MIPtool is the base — do NOT fold squid2minerva scope into these tickets.

Sub-tickets: 183 (one fov/well), 184 (output), 185 (easy view), 186 (CLI), 187 (4-fov design), 188 (throughput + pluggable projector), 189 (ingest), 192 (multi-acq, optional).

## Eng review — resolved decisions (2026-07-04)

These decisions were made during `/plan-eng-review` and are now binding for implementation.

1. **Packaging is the true keystone.** Add a foundational packaging ticket **before** IMA-189: `pyproject.toml`, `console_scripts` entry point for the CLI, pinned environment. Without it the tilefusion dependency has no delivery mechanism and acceptance #4 ("one-command batch") cannot be met. (D3)
2. **Consume tilefusion as a pinned git dependency** (`tilefusion @ git+<remote>@<commit-or-tag>`), not a local path (breaks off-machine/CI) and not vendored (forks the reader). Requires tilefusion to have a reachable remote and its own installable `pyproject`. (D4)
3. **Metadata reuse ≠ pixel reuse.** tilefusion is reused for **format recognition + metadata only**. Pixel reads bypass tilefusion (it casts to `float32` and averages RGB via `_to_grayscale_2d`) and use a raw per-plane read preserving exact `uint16`. This split is explicit in the diagram below and in the IMA-189 adapter contract — it is the epic's most likely correctness bug if left implicit. (D5)
4. **Normalize channels once, in the 189 adapter.** The adapter parses `acquisition_channels.yaml` and returns a normalized channel model (`name`, file-token, `display_color`, canonical order). Both output consumers (OME-zarr writer, per-well TIFF) consume it; neither re-parses. Kills the spaces-vs-underscores, nested-`display_color`, and descending-order bug class. (D6)
5. **FIJI-parity golden test is committed.** Generate FIJI `Z-Project(Max)` output for a small multi-channel crop of the real hongquan z-stack; commit input + expected; assert bit-exact. Doubles as the AC4 `uint16` guard. Add a synthetic unit test as a fast math complement. (D7)
6. **Projector streams a running max.** Read one z-plane per channel at a time, maintain a `np.maximum` accumulator; peak RAM is independent of `Nz`. Matches the architecture's "streaming" claim. NOTE: `tifffile.imread` loads a whole file, so streaming a multi-page OME-TIFF requires `TiffFile` page iteration, not `imread` — see the read spike below. (D8, D11)
7. **Reorder: 184 (output) before 188 (throughput).** Throughput can only be measured end-to-end; zarr/OME-TIFF write is a co-equal bottleneck. 188 becomes an optimization pass over the complete pipeline. (D9)
8. **Write display/contrast metadata into the OME-zarr.** Compute per-channel display windows (percentile or min/max) and write omero `window` fields alongside `display_color`. Raw `uint16` with no window renders near-black in ndviewer_light — "opens" (AC3) is not "easy to view" (the thesis). Flat-field correction is explicitly OUT of scope. (D10)
9. **Read-primitive spike before build.** Confirm across all three input formats (individual-TIFF, OME-TIFF-tiles, zarr) that you can get bit-exact `uint16` per plane AND iterate planes without loading the whole file. Document the per-format read strategy in the 189 spec. De-risks decisions 3 and 6 for formats beyond the tested individual-TIFF case. (D11)
10. **Reconfirm 1-FOV/well with Nick/You Yan** before committing the pipeline to single-FOV. If real 1536wp plates tile multiple FOVs/well, the MVP produces partial images and multi-FOV = stitching (a larger scope). FOV count stays a parameter regardless. (D12)

## Failure modes (per new codepath)

| Codepath | Realistic production failure | Test? | Error handling? | Silent? |
|---|---|---|---|---|
| Pixel read (uint16 split) | Reader wired to tilefusion's float32/RGB-average path → pixels corrupted | FIJI golden + bit-exact (decision 5) | N/A (correctness) | **Would be silent → CRITICAL if untested** |
| Channel normalization | Wrong color/order in one consumer, not the other (spaces/underscore, nested color, descending YAML) | Channel order/color assertion needed | Validate channel count vs metadata | Silent on smoke test → real-data bug |
| Streaming projector | `imread` loads whole multi-page OME-TIFF → OOM / defeats streaming | Peak-RSS test at scale (blocked on target) | Must use `TiffFile` page iter | Silent (slow/OOM) |
| Read primitive (OME-TIFF/zarr) | tilefusion returns abstract array, no addressable raw plane | Read spike (decision 9) | Per-format strategy documented | Would surface mid-build → spike de-risks |
| OME-zarr display metadata | Missing contrast window → near-black image | Visual check in ndviewer_light | Compute window per channel | AC3 passes, thesis fails → user-visible |
| Worker pool memory | Worker count × per-worker footprint blows RAM on many-core box | Bounded-memory test | RAM-aware worker cap | Silent (OOM) until scale |

**Critical gap:** the uint16 pixel-read path is silent-if-wrong and correctness-critical — the committed FIJI golden test (decision 5) is the mandatory guard. No other failure mode is both silent AND untested after the decisions above.

## Second-pass additive findings (2026-07-04, verified against tilefusion/ndviewer_light source)

A second `/plan-eng-review` run read the reused source directly and confirmed the decisions above; it **corroborates D5** — `individual_tiffs.py:308` ends `return stacked.astype(np.float32)` and `_to_grayscale_2d` averages RGB→luminance, so the bypass-tilefusion-for-pixels / raw-`uint16` read is the correct path (this also makes native-`uint16` max accumulation automatic). The following are additive and not covered above:

1. **`well_index` adapter (correctness-critical).** The reader is **tile-indexed**: metadata carries `tile_identifiers` as a flat list of `(well, fov)` tuples, not a well/fov API. Build a dedicated, tested adapter: group by well → `well → [(fov, tile_idx)]`; handle all 3 filename conventions (region+fov, region-only, manual/fov-only); **reject manual/fov-only ("not a well-plate") with a clear error**. This is the seam a silent well-swap bug hides in. → adds to the 189/183 boundary. (T10)
2. **`coordinates.csv` z-indexing gate (blocks the projection loop).** `individual_tiffs.py:164` filters positions to `z_level==0` **only if a `z_level` column exists**; otherwise every CSV row counts as z=0. If Squid writes one row per (fov, z) with no `z_level` column, `tile_identifiers` **inflates by Nz** and the loop mis-indexes. Verify against one real acquisition before building. Folds into the read-primitive spike (T2). (T11)
3. **Partial/ragged-plate semantics.** `n_z` is global (from `acquisition parameters.json`), so an aborted/short stack is **not** detectable from metadata — it surfaces as a per-read `FileNotFoundError`. Guard each read; skip-and-log missing/short wells; validate channel presence; emit a projected-vs-skipped summary. Per-well independence so one bad well doesn't abort the batch. (T12)
4. **`plate.zarr` layout must match ndviewer_light exactly.** `ndviewer_light/core.py:845-874,1183-1201` walks `plate.zarr/{row}/{col}/{field}/…`, `well_id = row_dir.name + col_dir.name`, and colorizes from `omero.channels[].color`. The region string (`"D5"`) must be split into row/col; **1536wp uses double-letter rows (AA–AF)**, so a naive `region[0]/region[1:]` split breaks. Add a round-trip test that opens the writer's output in ndviewer's loader. Strengthens D10 (contrast window) with the layout + double-letter requirement. (T13)
5. **Explicit input scope.** A Squid acquisition saved directly as `plate.zarr` carries `_squid`, not `per_index_metadata`, so `open_reader` (`base.py:341`) falls through to `_IndividualTiffsReader` and crashes on missing `coordinates.csv`. **Declare "individual-TIFF / OME-TIFF-tiles input only" for now**; guard zarr input with a clear "unsupported input" error.
