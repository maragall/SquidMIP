# IMA-229 — Zarr input

_slug: zarr-read_

## Why
`reader.py` raises `NotImplementedError` today (reader.py:106-109). Squid can save an
acquisition as Zarr v3 instead of individual TIFFs, and those acquisitions currently cannot
enter the MIP pipeline at all.

## Scope
Read Squid **HCS (`plate.ome.zarr`) Zarr v3 acquisitions** through the same reader API as
individual TIFFs, so `open_reader()` → `select_fovs` → `project_plate` → `write_plate` works
unchanged. The two non-wellplate Zarr layouts are **detected and refused with a clear
message**, not implemented (see NOT in scope).

## Locked decisions (from /plan-eng-review, 2026-07-20)

Decisions D1-D8 were taken interactively; **D1, D2 and D3 were then corrected and D9 added**
after the outside-voice pass verified three of them against Squid and ndviewer_light source.
The corrections are recorded in place, with the superseded reasoning marked, so nobody
re-derives the wrong version.

**D1 — HCS only; fix detection.** Implement `plate.ome.zarr`. Extend the detection branch so
per-FOV (`zarr/{region}/fov_*.ome.zarr`) and 6-D (`zarr/{region}/acquisition.zarr`) layouts are
recognised and refused **by name**. Today they match neither `path/"zarr.json"` nor
`path.glob("*.zarr")` (the directory is named `zarr`, not `*.zarr`), so they fall through to
`SquidReader` and die with a misleading `"No Squid individual-TIFF files found"`.

> **CORRECTED.** The original rationale — "those layouts exist for non-wellplate acquisitions,
> and `_output.py:90-95` refuses non-well-shaped region ids anyway" — is **false**.
> `multi_point_worker.py:268` selects the layout from the acquisition MODE
> (`is_hcs = is_select_wells or is_loaded_wells`), not from region naming. A genuine wellplate
> run in flexible/manual-region mode produces the per-FOV layout with region ids like `B2`,
> which `parse_well_id` accepts and `write_plate` would handle fine.
> The scope decision (HCS only) **stands** on effort/priority grounds, but two things follow:
> the refusal message must NOT claim "non-wellplate", and real users with real plates WILL hit
> it. Per-FOV support is now a TODO with a concrete trigger, not a hypothetical.

**D2 — Viewer uses `start_zarr_acquisition()`, and the reader exposes a positive capability.**
`plane_ref` promises `(filepath, page_index)` (reader.py:242-245) and
`ndviewer_light.register_image` documents `page_idx` as a TIFF page (`core.py:2294-2317`).
A zarr plane has neither. `SquidZarrReader` therefore exposes:
- `supports_plane_ref = False`
- `fov_store_path(region, fov) -> str` — the per-field group directory

and `_viewer.py` calls
`start_zarr_acquisition(fov_paths, channels, num_z, fov_labels, height, width)`
(`ndviewer_light/core.py:2788`) — the same seam Squid's own HCS GUI uses.

> **CORRECTED.** The original decision was `load_dataset(acquisition_path)`. That is the wrong
> API: `load_dataset` (core.py:3710) never sets `_fov_labels`, and `go_to_well_fov` returns
> `False` immediately when `_fov_labels` is empty (core.py:2511). Double-click navigation and
> the plate↔slider red-box link (`_viewer.py:1906` reads `_detail._fov_labels`) would both
> silently stop working. `load_dataset` would also impose ndviewer's own
> `discover_zarr_v3_fovs` ordering over SquidMIP's `_plate_key` row-major `order`, desynchronising
> the slider index from `_fov_index[well]["idx"]`, which `_on_push` / `_on_preview_tile` depend on.
> `start_zarr_acquisition` takes `fov_labels` explicitly, so SquidMIP keeps control of ordering.

Critical detail either way: simply raising `NotImplementedError` from `plane_ref` would **crash
the GUI** — `_viewer.py:1893` catches only `(KeyError, IndexError, OSError, RuntimeError)` and
`_viewer.py:1613` catches only `(KeyError, IndexError, OSError)`.

**D3 — Refuse incomplete acquisitions by default; treat metadata as a plan, not a record.**
A zarr array is allocated full-size up front; unwritten chunks decode as the fill value (zeros),
so a crashed or aborted run projects to a plausible-looking but **wrong** plate. The
individual-TIFF path has no such hole (a missing plane is a missing file → `KeyError` at
reader.py:217). Read `_squid.acquisition_complete`; if false, raise. Both the CLI
(`--allow-incomplete`) and the **GUI open path** must offer the override.

> **CORRECTED — this is the most consequential fix.** Plate and well metadata are written
> **up front from the acquisition plan**, on the first frame, not accumulated as wells finish
> (`job_processing.py:549-570`: `wells` comes from `get_hcs_structure()`, and
> `fields = list(range(fov_count))`). So on a partial plate the metadata **over-claims**: it
> lists wells and fields whose directories do not exist on disk at all.
> Two consequences the original decision got backwards:
> 1. Discovery must **intersect** metadata with what exists on disk (see the diagram above).
>    The original "prefer metadata, fall back to dir-walk when truncated" guards the wrong
>    direction entirely — truncation is not the failure mode; over-claiming is.
> 2. Without that intersection, `allow_incomplete=True` — the escape hatch that exists
>    precisely for partial plates — would hit `FileNotFoundError` on a planned-but-absent
>    `{row}/{col}/{fov}/zarr.json` instead of warning and proceeding. **The escape hatch would
>    not work on the only case it exists for.**

There are **three** end states, not two, and the error message must distinguish them:

| State | `acquisition_complete` | `aborted` | Directories |
|---|---|---|---|
| Clean finish | `true` | absent | all present |
| User abort | `false` | `true` | wells up to the abort point |
| Hard crash / power loss | `false` | absent | wells up to the crash |

**D4 — `omero.channels[].label` is the channel ground truth.** It is written by iterating the
C axis (`zarr_writer.py:544`), so its order *is* the array's C order. `acquisition_channels.yaml`
supplies display colors only, via the existing `resolve_channels()`. Warn on disagreement.
Same principle as IMA-189 trusting filenames over `coordinates.csv`. Deliberately diverges from
`SquidOMEReader`'s yaml-first order (reader.py:324-332) — document why, in place.

**D5 — `acquisition.yaml` wins for scalar physical metadata.** `pixel_size_um`, `dz_um` and
`wellplate_format` come from `load_acquisition_metadata()` as they do for every other reader
(`_acquisition.py:3-5,62`). `_squid.pixel_size_um` / `_squid.z_step_um` are read only as a
cross-check that `warnings.warn` on drift, mirroring the existing Nz/Nt checks at reader.py:179-188.
`acquisition.yaml` stays REQUIRED for zarr — `wellplate_format` exists nowhere else and
`_output.py` needs it.
**Amendment:** `wellplate_format` can legitimately be **present-but-null**
(`getattr(scanCoordinates, "format", None)`). Handle `None` explicitly rather than assuming the
key's presence means a usable value.

**D6 — Extract shared metadata helpers first, in a separate commit — but write
characterization tests BEFORE the refactor.** `SquidReader.metadata` (reader.py:163-208) and
`SquidOMEReader.metadata` (:319-350) already duplicate the region/fov assembly, the eleven-key
metadata dict, and the Nz cross-check warning. A third copy is the drift risk.

> **AMENDED.** The claimed regression gate — "the existing 20 tests must stay green" — is
> thinner than it looks. Exactly **one** of those 20 tests touches `SquidOMEReader.metadata`
> (test_reader.py:197), and it uses a single well, a single FOV and a matching yaml. Nothing
> covers the yaml/`n_c` mismatch fallback, `_ome_channel_names`, multi-region `_plate_key`
> ordering, or the Nz warning on the OME path. T1 would refactor code the suite does not cover.
> **Add OME-reader characterization tests first, or the gate is theater.**

**D7 — Rewrite the module docstring and its diagram.** reader.py:1-6 still claims OME-TIFF and
Zarr are "detected and rejected" while `SquidOMEReader` is implemented at :267 — already false
before this ticket. Replace the single-path flow diagram (:17-24) with the dispatch tree above
and give `SquidZarrReader` its own layout diagram.

**D8 — Validate dtype at open, keep the per-plane guard.** A zarr array's dtype is in its
`zarr.json`, so check it against `_SUPPORTED_DTYPES` once when the store opens rather than
discovering it on plane 1000 of a plate run. Squid explicitly contemplates float stores
(`zarr_writer.py:571-577`). Keep `_validate_plane` on each returned slice as a backstop against a
store whose chunks disagree with its header.

**D9 (NEW) — Dispatch must distinguish a Squid acquisition from SquidMIP's own output.**
`write_plate` writes `<out>/plate.ome.zarr` (`_output.py:301`) — structurally identical to what
Squid writes. After D1's detection fix, pointing `open_reader` at our own output directory
returns a `SquidZarrReader` that then dies on a missing `acquisition.yaml` with an error about
the wrong thing entirely. The discriminator is free and already documented above: **Squid writes
a `_squid` block and no `field_count`; we write `field_count` and no `_squid`.** Refuse our own
output with a message that says so.

## Implementation sketch

```python
class SquidZarrReader:
    """Lazy reader over a Squid HCS Zarr v3 acquisition ({acq}/plate.ome.zarr).

    Same interface as SquidReader (metadata + read), minus plane_ref: a zarr plane is a
    slice of a chunked array, not a (file, page). Instead it exposes fov_store_path(), which
    the viewer hands to ndviewer_light.start_zarr_acquisition().

    Plate and well group metadata describe the acquisition PLAN, not what was acquired
    (job_processing.py:549-570 writes them up front), so discovery INTERSECTS them with the
    directories that actually exist. One tensorstore handle per field, opened on demand and
    cached; read() slices [t, c, z, :, :].
    """
    supports_plane_ref = False

    def fov_store_path(self, region, fov) -> str: ...
```

- Open with the spec already used at `_montage.py:78-80`:
  `ts.open({"driver": "zarr3", "kvstore": {"driver": "file", "path": ...}}, open=True)`.
- Resolve the array subdirectory from `ome.multiscales.datasets[0].path` rather than
  hardcoding `"0"`.
- Never require `field_count`: Squid does not write it (and its presence means the opposite —
  see D9).
- `region = row_name + col_name` by concatenation, which round-trips through
  `_output.parse_well_id` because Squid writes columns unpadded.
- Regions sorted with the existing `_plate_key` so plate order stays row-major.
- **Memory note:** one plane read decompresses one chunk under NONE/FAST, where chunks are
  `(1,1,1,y,x)`. Under BALANCED/BEST the shard is `(1, c_size, 1, Y, X)` — all channels for a
  `(t,z)` — so a per-channel-per-z projection re-touches each shard once per channel. Not fatal
  and not a blocker, but do not claim "exactly one plane decompresses per call" in comments.

- `_plate_key` (reader.py:74-87) — row-major plate ordering, used as-is.

## NOT in scope

- **Per-FOV zarr (`zarr/{region}/fov_*.ome.zarr`)** — deferred on effort/priority, **not**
  because it is non-wellplate (that rationale was wrong; see D1). Real wellplate users in
  flexible-region mode will hit the refusal. Captured as a TODO with a concrete trigger.
- **6-D zarr (`acquisition.zarr`, `(FOV,T,C,Z,Y,X)`)** — Squid's own docs call it non-standard
  and warn most OME-NGFF viewers misread it (`zarr-v3-format.md:61`). Refused by name.
- **Zarr v2** — Squid writes v3 only.
- **Multi-page TIFF (`{region}_{fov}_stack.tiff`)** — still unimplemented, unchanged (reader.py:143).
- **Live/streaming read of an in-progress acquisition** — `allow_incomplete` opens a snapshot;
  it does not follow a running acquisition.
- **Brightfield / RGB zarr** — same deferral as the TIFF path (see TODOS.md).
- **Distribution** — no new artifact; same package, same console scripts.

## Acceptance / oracle
1. A Squid HCS `plate.ome.zarr` acquisition loads through `open_reader()` and produces the same
   eleven-key `metadata` contract as the individual-TIFF reader.
2. `open_reader(zarr_acq)` → `select_fovs` → `project_plate` → `write_plate` completes and the
   written plate validates under `tests/ngff_check.assert_valid_ngff_plate`.
